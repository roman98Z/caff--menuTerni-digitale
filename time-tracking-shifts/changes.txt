Changes / design notes
======================

This file documents every implementation choice that goes beyond what was
explicitly required by the specification, as required by §6 "Libertà
implementative".

General architecture
--------------------
* Layered layout under ``app/`` with dedicated sub-packages:
  ``app/routers`` (HTTP), ``app/algorithms`` (domain), plus cross-cutting
  modules (``config.py``, ``database.py``, ``security.py``,
  ``dependencies.py``, ``exceptions.py``, ``utils.py``).
  Reason: keeps each file focused and testable, makes it obvious where to
  extend.
* Single ``create_app()`` factory in ``app/main.py`` so the app can be built
  programmatically for tests / embedded use.
* Pydantic v2 schemas are split into explicit `*Create`, `*Update`, `*Out`
  classes (no ORM leaks into the API layer).

Database
--------
* A custom ``Point`` ``UserDefinedType`` is introduced in ``app/models.py``
  that round-trips asyncpg's native ``asyncpg.Point`` into a plain
  ``(lat, lon)`` tuple. Reason: the spec requires the Postgres ``POINT`` type
  but SQLAlchemy has no built-in mapping for it.
* Added PostgreSQL indexes that were not explicitly required:
  - ``ix_users_service_id``, ``ix_users_active``
  - ``ix_entries_user_id``, ``ix_entries_datetime``
  - ``ix_shifts_user_id``, ``ix_shifts_start_time``
  - ``ix_user_constraints_user_id``, ``ix_user_preferences_user_id``
  - ``ix_shift_templates_service_id``
  Reason: these columns are used by almost every query we run (date-range
  reporting, per-user listings, joins). The overhead on write is negligible
  for a PMI dataset.
* ``entries.datetime`` is kept as ``TIMESTAMPTZ`` (as required) but
  ``shifts.*``, ``user_constraints.*`` and ``shift_requirements.*`` use plain
  ``TIMESTAMP`` because those tables describe local working hours (the spec
  uses the unqualified "TIMESTAMP" keyword for them).
* ``shift_requirements.service_id`` was **added** (nullable, cascades on
  service deletion) so that each service can have its own staffing targets.
  The spec only defined the generic columns; without scoping, all services
  would share the same requirements set.

Security
--------
* Passwords auto-generated by ``POST /services/{id}/users`` are exactly 8
  characters long, always contain at least one uppercase letter and one
  special character from ``!@#$%^&*``. The generator rejects any sample that
  doesn't satisfy those rules. The plaintext password is only returned once
  (in the creation response) and stored as SHA-256.
* ``security.verify_hash`` uses ``hmac.compare_digest`` to avoid timing
  attacks.
* SHA-256 is used for both password_hash and passkey storage *as required by
  the spec*. For real production deployments a rinforzato scheme such as
  argon2id / bcrypt is strongly recommended; this is documented in README.md.
* The passkey for ``create_service.py`` is generated with
  ``secrets.token_urlsafe(32)`` (URL-safe 256-bit token).

Authentication model
--------------------
* Two FastAPI dependencies centralize the check:
  - ``require_service_passkey`` for routes under ``/services/{service_id}/…``
  - ``service_from_user`` for routes addressed to a user but requiring the
    *service* passkey (e.g. ``PATCH /users/{user_id}``)
  - ``require_user_passkey`` for checkin/checkout + personal queries
  - ``require_user_or_service_passkey`` for ``GET /users/{user_id}/shifts``
    (either passkey is accepted)
* Passkey accepted via header ``X-Passkey`` or ``?passkey=`` query param.
  Rationale: tenderers can use whichever is easiest for their client without
  losing compatibility.
* ``DELETE /entries/{entry_id}``, ``PUT/POST/DELETE /shifts`` require the
  service passkey via ``?service_id=…``.  The shift-scope endpoints receive
  ``service_id`` explicitly because the Shift table carries no service FK
  (only via users.service_id).

Entries / late-early detection
------------------------------
* Each GET entries endpoint returns an enriched payload:
  ``{ id, user_id, datetime, type, geoloc, late, early,
      related_shift_id, delta_minutes }``.
  The ``late`` / ``early`` flags are computed against the nearest planned
  shift for the same user in the relevant date range. A fixed tolerance of
  5 minutes is applied; configurable by tweaking
  ``match_entries_to_shifts(tolerance_minutes=...)``.
* If the user has no planned shift in the window, ``late = early = false``
  and ``related_shift_id = null``.

Scheduling algorithms
---------------------
* All three algorithms share:
  - a common ``Slot`` representation
  - a common ``UserProfile`` that caches effective constraints/preferences
  - the same hard-feasibility check (``is_feasible``)
* ``ilp`` falls back to ``greedy`` automatically if PuLP is not installed or
  the solver fails to reach a feasible solution within 30 seconds.
* ``genetic`` is *usable* without DEAP: the core loop is a compact custom GA
  (tournament selection, uniform crossover, triple-operator mutation). DEAP is
  imported only to mark that stats registries are available — this keeps the
  dependency optional for low-footprint deployments.
* The scheduler works off ``shift_requirements`` first; if none are defined,
  it materializes ``shift_templates`` across the horizon with a default
  ``required_count = 1``. Documented here because the spec did not describe
  this fallback behaviour.
* Results are *never* persisted automatically: ``POST /services/.../shifts/generate``
  returns a preview unless ``commit=true`` is passed in the body.

Operational choices
-------------------
* Added middleware with:
  - structured logging of method / path / status / latency / client IP
  - simple sliding-window rate limiter (per-IP, env-configurable, default
    120 req/min, ``0`` disables it)
  Reason: sensible production defaults for a self-hosted PMI service.
* Global exception handlers translate ``AppError``, SQLAlchemy
  ``IntegrityError`` and generic ``SQLAlchemyError`` into JSON responses.
* ``ENV`` configuration via ``pydantic-settings`` (``.env`` loaded
  automatically). ``.env.example`` provided.
* CORS: allows ``*`` by default, overridable via ``CORS_ORIGINS``
  (comma-separated list).

Deliverable
-----------
* The project ships with BOTH an Alembic migration (``alembic/versions/0001_initial.py``)
  AND a raw SQL init script (``sql/init.sql``). Either can be used; the
  README documents both paths.
* ``requirements.txt`` pins *minimum* versions only (``>=``); pinning exact
  versions is left to the deployer (``pip freeze > requirements-lock.txt``).
